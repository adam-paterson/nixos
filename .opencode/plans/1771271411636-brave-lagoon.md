# Developer Experience Plan (Snowfall shells + devenv)

## Goals
- Keep `src/shells/*` as the source of truth, with devenv tasks as canonical task logic.
- Add a minimal `ci` shell for deterministic CI execution and reduced tool drift.
- Restore a usable `Justfile` as an easy command surface over nix/devenv tasks.
- Add containerized Ubuntu validation focused on building NixOS artifacts.

## Recommended Implementation

### 1) Stabilize baseline and remove DX drift
- Ensure docs and automation reflect the real shell layout (`src/shells/dev/default.nix`).
- Stop tracking machine-generated devenv artifacts and keep them ignored.
- Confirm flake outputs for both dev and CI shells are visible and evaluable.

Critical files:
- `flake.nix`
- `.gitignore`
- `README.md`

### 2) Add a minimal CI shell under Snowfall shells
- Create `src/shells/ci/default.nix` using `inputs.devenv.lib.mkShell`.
- Keep `ci` intentionally minimal (only tools needed for formatting, linting, eval/check/build).
- Keep `src/shells/dev/default.nix` for interactive tooling.
- Optionally factor shared shell bits into `src/shells/common/default.nix` to avoid drift.

Critical files:
- `src/shells/ci/default.nix` (new)
- `src/shells/dev/default.nix`
- `flake.nix`
- `src/shells/common/default.nix` (optional new)

### 3) Canonical tasks in devenv; Justfile as wrappers
- Define/complete canonical tasks in shell config (format, lint, check, eval, cache-targets, ci/fix flows).
- Populate `Justfile` with thin wrappers only (no duplicated business logic).
- Keep task names stable for ergonomics and CI continuity (`fmt-check`, `lint`, `check`, `eval`, `fix`, `ci`, `cache-targets-linux`, `cache-targets-macos`).

Critical files:
- `src/shells/dev/default.nix`
- `src/shells/ci/default.nix`
- `Justfile`

### 4) CI wiring to use deterministic shell/task entrypoints
- Update `nix-checks.yml` to run via `.#ci` shell and task wrappers.
- Keep CI commands explicit and deterministic (`nix develop .#ci -c just ...` or equivalent wrapper path).
- Preserve cache/deploy workflows, but align them with the new shell/task contract where relevant.

Critical files:
- `.github/workflows/nix-checks.yml`
- `.github/workflows/nix-cache.yml` (if task command alignment is needed)
- `.github/workflows/deploy-aurora.yml` (if command references need alignment)

### 5) Ubuntu container validation for NixOS artifact builds
- Expand current container strategy so CI can validate artifact builds in an Ubuntu container context.
- First milestone: build one representative artifact in containerized Ubuntu (e.g. `.#nixosConfigurations.aurora.config.system.build.toplevel`).
- Add follow-up targets incrementally after first stable pass.

Critical files:
- `Containerfile`
- `.github/workflows/ubuntu-artifacts.yml` (new) or extend existing checks workflow
- `Justfile` / shell tasks (for a single command entrypoint)

### 6) Documentation and migration guardrails
- Update README command contract to match real commands and shell paths.
- Document `dev` vs `ci` shell purpose and when to use each.
- Add a short migration note so old task habits are mapped to new commands.

Critical files:
- `README.md`
- `Justfile`

## Rollout Order
1. Baseline hygiene (tracked generated file policy + docs path correction).
2. Add `ci` shell.
3. Implement canonical tasks and Just wrappers.
4. Switch `nix-checks.yml` to CI shell/task contract.
5. Add Ubuntu container artifact build validation.
6. Final docs cleanup.

## Verification Plan

### Local
- `nix flake show`
- `nix develop .#dev`
- `nix develop .#ci`
- `nix run nixpkgs#just -- fmt-check`
- `nix run nixpkgs#just -- lint`
- `nix run nixpkgs#just -- check`
- `nix run nixpkgs#just -- eval`
- `nix run nixpkgs#just -- ci`

### Container/Ubuntu validation
- `docker build -f Containerfile -t nix-ci .`
- `docker run --rm -v "$PWD:/work" -w /work nix-ci nix build .#nixosConfigurations.aurora.config.system.build.toplevel`

### CI acceptance criteria
- `Nix Checks` passes using the new CI shell/task entrypoint.
- Ubuntu container artifact job passes for the selected target.
- Cache/deploy workflows continue to resolve expected build attributes.

## Risks and Mitigations
- **Task drift between dev and CI:** keep canonical task logic in shell config; Justfile wrappers only.
- **Generated devenv file churn:** keep generated artifacts out of git.
- **Platform-specific breakage:** validate Linux targets in Ubuntu container first, then widen scope.
- **Workflow regression risk:** migrate checks workflow first, then cache/deploy adjustments in follow-up.
